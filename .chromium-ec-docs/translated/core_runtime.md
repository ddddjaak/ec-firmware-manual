# Chromium OS 嵌入式控制器运行时

## 设计原则

1. **能在编译时做的事绝不在运行时做** — 目标是节省闪存空间和计算量。编译时配置优先，除非确实需要运行时切换。

2. **实时性：保证低延迟（如 < 20μs）** — 中断处理程序中不禁止中断，代码有界。

3. **保持简单** — 针对我们使用的微控制器子集设计，面向 32 位单核 CPU 的小型系统：4kB 到 64kB 数据 RAM，可能从 Flash 就地执行。

## 执行上下文

这是一个可抢占的运行时，具有静态任务。只有 2 种可能的执行上下文：

- 常规**任务**
- **中断处理程序**

初始启动是一个例外（参见启动部分）。

### 任务

任务在编译时静态定义。每个板级的任务在 `board/<board>/ec.tasklist` 文件中描述。

任务的优先级由其在 `ec.tasklist` 文件中的顺序隐式定义（最上面的优先级最低，即任务 1）。因此，两个不同的任务不能有相同的优先级。

每个任务有自己的栈，栈大小在 `ec.tasklist` 文件中编译时定义。

任务可以在任何时候被中断或更高优先级的任务抢占。

### 中断

硬件中断请求通过 `DECLARE_IRQ` 宏声明的 C 中断处理程序连接。

中断可以嵌套（即被更高优先级的中断打断）。所有中断向量在 `DECLARE_IRQ` 宏中分配优先级。中断处理程序只能被**严格更高**优先级的处理程序打断。

### 事件

每个任务有一个待处理事件位图，实现为 32 位字。几个事件对所有任务预定义，最高有效位保留：

- 位 31：定时器待处理事件
- 位 29：请求任务唤醒
- 位 30：互斥锁等待者唤醒

最低 19 位可用于任务特定含义。

事件位用于任务间通信和调度机制。其他任务和中断处理程序可以原子地设置它们来请求特定操作。使用 `task_set_event()` 和 `task_wake()` 原语。

典型用例：

- 一个任务向另一个任务发送消息（使用共享内存结构），要求其立即处理。
- 硬件中断发生，需要长时间处理（如 I2C 事务）。关联的中断处理程序因延迟原因无法完成，因此触发事件要求任务完成。

任务通过 `task_wait_event()` 和 `task_wait_event_mask()` 原语选择消费事件。

### 调度与抢占

系统有一个全局位图 `tasks_ready`，每个任务一位，指示是否"准备好运行"。调度策略仅仅是找到 `tasks_ready` 位图中最高有效位并调度对应任务。

重要说明：重新调度**仅在退出中断上下文时**发生。有 3 种情况触发重新调度：

1. 中断处理程序为任务设置新事件 — `task_set_event` 检测到在中断上下文中执行，记录需要在中断返回时重新调度。
2. 一个任务为另一个任务设置事件 — 运行时触发软件中断以强制重新调度。
3. 运行的任务主动调用 `task_wait_event()` 放弃执行权。

### Hooks 与延迟函数

最低优先级的任务（任务 1，即 TASK_ID_HOOKS）用于执行重复操作和延迟函数，无需阻塞当前任务或创建专用任务。

HOOKS 任务维护一个延迟函数及其下次截止时间的列表。每次唤醒时，它遍历列表并调用截止时间已过的函数。延迟函数使用 `DECLARE_DEFERRED()` 宏创建。HOOK_SECOND 和 HOOK_TICK 钩子由 HOOKS 任务循环周期性调用。

!!! warning "优先级反转"
    在延迟函数中访问受互斥锁保护的资源（如共享 I2C 控制器）时要特别注意优先级反转。作为最低优先级任务，它可能被长时间抢占。

!!! warning "长延迟"
    在 Hook 或延迟函数处理程序中避免长延迟（> 100μs），因为这会饿死其他 Hook 的执行时间。最好实现状态机来安排后续调用。

### 看门狗

系统始终受到硬件看门狗的保护，当任务异常时会重启 CPU。

看门狗在 HOOKS 任务中喂狗，通常通过声明定期执行的 HOOK_TICK 来实现。由于这是最低优先级的任务，这保证了所有任务在看门狗周期内都能获得一些运行时间。

!!! note
    不要在代码中到处添加 `watchdog_reload()` 来掩盖长时间运行的问题。

为帮助调试触发看门狗重启的异常序列，大多数平台实现了 `CONFIG_WATCHDOG_HELP` 定义的警告机制。它是在看门狗周期中点触发的定时器，如果看门狗未被喂狗，则转储当前执行状态。

### 启动

启动序列经过以下步骤：

1. 汇编入口程序清除 .bss（未初始化数据），复制初始化数据，设置栈指针。
2. 跳转到 `main()` C 函数。
3. 硬件预初始化和初始化：内存保护、GPIO 默认状态、准备中断控制器、设置时钟、定时器、使能中断、初始化调试 UART 和看门狗。
4. 启动任务。

任务启动时，最初只有 HOOKS 任务标记为就绪，因此它首先启动并调用所有 HOOK_INIT 处理程序执行初始化。然后所有任务标记为就绪，最高优先级任务获得控制权。

### 锁定与原子性

两个主要的并发原语是轻量级原子变量和较重的互斥锁。

- **原子变量**：32 位整数，`atomic.h` 头文件包含原子位操作和算术操作原语。
- **互斥锁**：静态分配的二值信号量。竞争时使等待任务休眠，解锁时使用事件机制唤醒其他等待者。

!!! note
    互斥锁不触发优先级提升来避免优先级反转现象。

## 时间

### 时间保持

运行时中，时间使用自 MCU **冷启动**以来的 **64 位微秒**计数。运行时没有挂钟/日期概念。

微秒时间戳使用 `timestamp_t` 类型，当前时间戳由 `get_time()` 函数返回。

### 定时器事件

运行时为每个任务提供**一个**（仅一个）定时器。所有任务定时器在单个硬件定时器上多路复用。每次定时器设置或到期时，运行时找到截止时间最近的任务定时器并编程到硬件中。

!!! note
    由于每个任务只有**一个**定时器，且 `task_wait_event()` 带超时调用也使用此定时器，直接使用 `timer_arm()` 函数时要小心。

## 内存

### 单地址空间

任务之间没有内存隔离（所有任务在同一地址空间中）。所有内存隐式共享，任务间通信可以通过简单地写入内存结构并使用事件唤醒其他任务完成。

### 堆

数据结构应在编译时静态分配。

!!! note
    没有动态分配器（如 `malloc()`），不是因为无法创建，而是为了避免负面影响：不良/不可预测的实时行为和可能的内存泄漏。

### 栈

每个任务有自己的栈，此外还有一个用于启动和中断/异常的系统栈。

!!! note "注意 1"
    每个任务栈相对较小（如 512 字节），实现功能时需注意栈使用。

!!! note "注意 2"
    栈使用的总 RAM 是总 RAM 消耗的很大一部分，因此需要仔细调整大小。
