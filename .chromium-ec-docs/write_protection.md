href="#Terminology"><span></span></a><a class="h" name="terminology" href="#terminology"><span></span></a>Terminology</h2><h2><a class="h" name="RO-and-RW" href="#RO-and-RW"><span></span></a><a class="h" name="ro-and-rw" href="#ro-and-rw"><span></span></a>RO and RW</h2><p>MCUs running the EC code have read-only (RO) and read-write (RW) firmware. Coming out of reset, the MCU boots into its RO firmware.</p><p>In the case of the EC, the RO firmware boots the host and asks it to verify a hash of the RW firmware (software sync). If the RW firmware is invalid, it is updated from a copy in the host&#39;s RW firmware.</p><p>In the case of the FPMCU, the RO firmware uses the public key embedded in it to validate the signature of the RW firmware. If the RW firmware is invalid it does not jump to the RW firmware.</p><p>Once the RW firmware is validated, the MCU jumps to it (without rebooting). The RO firmware is locked in the factory and is never changed. The RW firmware can be updated later by pushing a new system firmware containing an updated RW region.</p><p>Note that both the RO and RW firmware regions are normally protected once write protect has been turned on.</p><p>In the case of the EC, the RW region is unprotected at MCU boot until it has been verified by the host. The RW region is protected before the Linux kernel is loaded.</p><p>In the case of the FPMCU, the RW region is protected before jumping the RO firmware jumps to it.</p><h2><a class="h" name="Hardware-Write-Protect-hw_wp" href="#Hardware-Write-Protect-hw_wp"><span></span></a><a class="h" name="hardware-write-protect-hw_wp" href="#hardware-write-protect-hw_wp"><span></span></a>Hardware Write Protect {#hw_wp}</h2><p>On modern Chrome OS devices, the Cr50 (aka GSC / TPM) provides a &ldquo;hardware write protect&rdquo; GPIO that is connected to the AP SPI flash, EC SPI flash, EEPROM, and FPMCU via a <a href="https://chromium.googlesource.com/chromiumos/platform/ec/+/aaba1d5efd51082d143ce2ac64e6caf9cb14d5e5/include/ec_commands.h#1599">GPIO</a>. This &ldquo;hardware write protect&rdquo; can only be disabled with <a href="https://chromium.googlesource.com/chromiumos/third_party/hdctools/+/HEAD/README.md">Servo</a> or <a href="https://chromium.googlesource.com/chromiumos/third_party/hdctools/+/HEAD/docs/ccd.md#SuzyQ-SuzyQable">SuzyQ</a>(<a href="https://chromium.googlesource.com/chromiumos/platform/ec/+/cr50_stab/docs/case_closed_debugging_gsc.md#ccd-open">&ldquo;CCD open&rdquo;</a>) and corresponds to <a href="https://chromium.googlesource.com/chromiumos/platform/ec/+/cr50_stab/docs/case_closed_debugging_gsc.md"><code class="code">OverrideWP</code></a> in ccd. Disabling this write protect disables it for everything connected to this signal.</p><p>In the case of the FPMCU, the hardware write protect GPIO is tied to the STM32 <code class="code">BOOT0</code> pin, which is what tells the MCU to enter the STM32 bootloader mode.</p><p>You may see various references to a <a href="https://www.chromium.org/chromium-os/firmware-porting-guide/firmware-ec-write-protection">write protect screw in documentation</a>. Older Chrome OS devices had a write protect screw that had to be physically removed. More details on this history can be found here: <a href="http://go/cros-wp-status">http://go/cros-wp-status</a>.</p><p>Another way of disabling hardware write protection is to remove the battery; this method is mainly used during bringup.</p><p>Additional reference: <a href="https://chromium.googlesource.com/chromiumos/docs/+/HEAD/write_protection.md">https://chromium.googlesource.com/chromiumos/docs/+/HEAD/write_protection.md</a></p><h2><a class="h" name="Changing-Hardware-Write-Protection" href="#Changing-Hardware-Write-Protection"><span></span></a><a class="h" name="changing-hardware-write-protection" href="#changing-hardware-write-protection"><span></span></a>Changing Hardware Write Protection</h2><p>Modifying the state of hardware write protection (via Cr50 GPIO) can be done if the <a href="https://chromium.googlesource.com/chromiumos/platform/ec/+/cr50_stab/docs/case_closed_debugging_gsc.md#ccd-open">&ldquo;CCD open&rdquo;</a> process has been completed.</p><div class="note"><code class="code">servod</code> <em>must</em> be running for <code class="code">dut-control</code> to work.
