href="#Where-to-add-tests"><span></span></a><a class="h" name="where-to-add-tests" href="#where-to-add-tests"><span></span></a>Where to add tests?</h2><p>Tests currently reside in <a href="https://source.chromium.org/chromiumos/chromiumos/codesearch/+/main:src/platform/ec/zephyr/test/">zephyr/test</a>. When adding a new test it&#39;s possible to either create a new directory which includes a <code class="code">testcase.yaml</code> or add a new entry to an existing <code class="code">testcase.yaml</code> if the test is a slight variation of an existing test.</p><h3><a class="h" name="How-to-decide-where-your-tests-should-go" href="#How-to-decide-where-your-tests-should-go"><span></span></a><a class="h" name="how-to-decide-where-your-tests-should-go" href="#how-to-decide-where-your-tests-should-go"><span></span></a>How to decide where your tests should go?</h3><p>If you&lsquo;re adding a new compilational unit, check your dependencies. Is this unit tied to an existing Kconfig? If so, it&rsquo;s possible that another test is already enabling it and linking your <code class="code">.c</code> file. It&#39;s probably worth your time to simply add a new test suite to that existing test binary, or at the very least, creating a variant of the binary by adding a new entry in the <code class="code">testcase.yaml</code> file.</p><p>If this doesn&lsquo;t work, it&rsquo;s possible that you might need to create a new test from scratch. First, decide if this will be an integration test or a unit test. Integration tests will build the full system (see below) and thus are sometimes easier to set up (since they look and feel like an application). For the same reason that makes them easier to set up, the lack of mocks can make things more difficult when trying to force execution down a specific code path.</p><p>In the case of integration tests, any test under <code class="code">zephyr/test</code> will serve as a good example. Alternatively, an example of a unit test can be found under <a href="https://source.chromium.org/chromiumos/chromiumos/codesearch/+/main:src/platform/ec/common/spi/flash_reg/">common/spi/flash_reg</a>.</p><h2><a class="h" name="The-structure-of-the" href="#The-structure-of-the"><span></span></a><a class="h" name="the-structure-of-the" href="#the-structure-of-the"><span></span></a>The structure of the <code class="code">testcase.yaml</code></h2><p>The top level <code class="code">testcase.yaml</code> uses the <code class="code">test:</code> block to define the attributes for each test case. The optional <code class="code">common:</code> block uses the same attributes as the test cases, but the attributes are applied to all test cases in the file. See <a href="https://docs.zephyrproject.org/latest/develop/test/twister.html#test-cases">here</a> for more details.</p><p>Some common attributes include:</p><ul><li><code class="code">extra_configs</code> - a list of Kconfigs to add to the test.</li><li><code class="code">extra_conf_files</code> - specifies a YAML list of additional Kconfig files to to apply to the build. Replaces the <code class="code">CONF_FILE</code> field in extra_args.</li><li><code class="code">extra_overlay_confs</code> - specifies a list of overlay Kconfig files. Replaces the <code class="code">OVERLAY_CONFIG</code> field in extra_args.</li><li><code class="code">extra_dtc_overlay_files</code> - specifies a list of additional device tree files to apply to the build. Replaces the <code class="code">DTC_OVERLAY_FILE</code> field in extra_args.</li></ul><p><code class="code">extra_args</code> is a string field that allows injecting free-form CMake variables into the build. It is rarely needed and the practice of specifying <code class="code">CONF_FILE</code>, <code class="code">OVERLAY_CONFIG</code>, or <code class="code">DTC_OVERLAY_FILE</code> here is deprecated. Please use the above fields for these functions, as they are much more readable.</p><h2><a class="h" name="Integration-tests" href="#Integration-tests"><span></span></a><a class="h" name="integration-tests" href="#integration-tests"><span></span></a>Integration tests</h2><p>Integration tests build the full EC.
