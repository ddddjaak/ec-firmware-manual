## Windows 平台环境搭建

### 安装 Chocolatey

Chocolatey 是 Windows 系统上的软件包管理器，让你能像在 Linux 中使用 apt-get 或 yum 一样，通过命令行快速安装、更新和管理各种软件。

以管理员身份打开 PowerShell，先确认执行策略：

```powershell
Get-ExecutionPolicy
```

> **这条命令做了什么：** 仅读取当前 PowerShell 会话的执行策略状态，不修改任何系统配置。PowerShell 执行策略决定了是否允许运行 `.ps1` 脚本文件，默认值 `Restricted` 表示禁止一切脚本执行。这条命令纯粹是「查看」，可以放心执行。

若返回 `Restricted`，执行以下命令：

```powershell
Set-ExecutionPolicy Bypass -Scope Process
```

> **这条命令做了什么：** 将当前 PowerShell **进程** 的执行策略临时设为 `Bypass`（允许运行脚本）。`-Scope Process` 是关键——这个修改**仅在当前终端窗口有效**，关闭窗口即失效，不会写入注册表，也不会影响其他终端、其他用户或系统全局策略。这是一种「临时开门，用完即关」的安全做法。

然后安装 Chocolatey：

```powershell
Set-ExecutionPolicy Bypass -Scope Process -Force; [System.Net.ServicePointManager]::SecurityProtocol = [System.Net.ServicePointManager]::SecurityProtocol -bor 3072; iex ((New-Object System.Net.WebClient).DownloadString('https://community.chocolatey.org/install.ps1'))
```

> **这条命令做了什么（逐步分解）：**
>
> | 分句 | 作用 | 影响范围 |
> |------|------|----------|
> | `Set-ExecutionPolicy Bypass -Scope Process -Force` | 临时放行脚本执行（同上条，`-Force` 跳过确认弹窗） | 仅当前进程 |
> | `[System.Net.ServicePointManager]::SecurityProtocol = ... -bor 3072` | 启用 TLS 1.2 加密协议（`3072` 即 `Tls12`） | 仅当前进程 |
> | `New-Object System.Net.WebClient` | 创建一个 HTTP 客户端对象，准备下载 | 无 |
> | `.DownloadString('https://...')` | 从 `community.chocolatey.org` 下载安装脚本内容到内存 | 网络连接（`community.chocolatey.org:443`） |
> | `iex (...)` | 在内存中执行下载的脚本 | 见下方详述 |
>
> **Chocolatey 安装脚本对系统的修改：**
> - 在 `C:\ProgramData\chocolatey\` 创建程序目录（约 20 MB）
> - 在 `C:\ProgramData\chocolatey\lib\` 存放后续安装的软件包
> - 将 `C:\ProgramData\chocolatey\bin` 添加到系统 PATH 环境变量
> - 在 `C:\ProgramData\chocolatey\bin\` 放置 `choco.exe` 主程序和相关 DLL
> - **不会**安装系统服务、**不会**添加开机自启项
> - 所有安装源均从 `community.chocolatey.org` 获取，由社区维护
>
> 如果你对执行远程脚本有顾虑，可以先在浏览器打开 [https://community.chocolatey.org/install.ps1](https://community.chocolatey.org/install.ps1) 查看完整脚本源码，确认后再执行。

安装成功后，终端输入 `choco` 或 `choco -?` 可查阅使用说明。

### 安装其他软件

以管理员身份打开 cmd.exe，禁用全局确认提示：

```powershell
choco feature enable -n allowGlobalConfirmation
```

> **这条命令做了什么：** 关闭 Chocolatey 安装软件时的交互式确认弹窗（`choco install xxx` 原本会弹出 `Do you want to run the script?` 这类提示）。这只是改变 Chocolatey 自身的行为偏好，存储在其内部配置（`C:\ProgramData\chocolatey\config\chocolatey.config` 中的 XML 文件），不涉及系统注册表或安全策略。

安装所需依赖项：

```powershell
choco install cmake --installargs 'ADD_CMAKE_TO_PATH=System'
choco install ninja gperf python311 git dtc-msys2 wget 7zip
```

> **每条命令做了什么：**
>
> | 命令 | 安装内容 | 对系统的修改 |
> |------|---------|-------------|
> | `choco install cmake --installargs 'ADD_CMAKE_TO_PATH=System'` | CMake 构建系统（来自 cmake.org） | 安装到 `C:\Program Files\CMake\`，`--installargs` 将 CMake 添加到**系统级** PATH（所有用户可用），写入注册表 `HKLM\SYSTEM\CurrentControlSet\Control\Session Manager\Environment\Path` |
> | `choco install ninja` | Ninja 构建加速器（来自 ninja-build.org） | 安装到 `C:\ProgramData\chocolatey\lib\ninja\tools\`，仅一个 `ninja.exe` 可执行文件，通过 Chocolatey 的 shim 机制加入 PATH |
> | `choco install gperf` | GNU gperf 完美哈希函数生成器 | 安装到 `C:\ProgramData\chocolatey\lib\gperf\tools\`，提供 `gperf.exe`，Chocolatey 自动加入 PATH |
> | `choco install python311` | Python 3.11（来自 python.org 官方安装包） | 安装到 `C:\Python311\`，添加 `C:\Python311\` 和 `C:\Python311\Scripts\` 到用户级 PATH，注册 `.py` 文件关联 |
> | `choco install git` | Git 版本控制（来自 git-scm.com 官方安装包） | 安装到 `C:\Program Files\Git\`，添加 `C:\Program Files\Git\cmd\` 到系统 PATH，注册右键菜单「Git Bash Here」 |
> | `choco install dtc-msys2` | Device Tree Compiler（MSYS2 移植版） | 安装 dtc.exe 到 Chocolatey 工具目录，用于编译/反编译 `.dts`/`.dtb` 设备树文件 |
> | `choco install wget` | GNU Wget 下载工具（Windows 移植版） | 安装 `wget.exe` 到 Chocolatey 工具目录，提供命令行文件下载能力 |
> | `choco install 7zip` | 7-Zip 压缩解压工具 | 安装到 `C:\Program Files\7-Zip\`，添加 PATH，注册 `.7z` 文件关联 |
>
> **安全说明：** Chocolatey 从各软件包的官方下载源（cmake.org、python.org、git-scm.com 等）获取安装包，而非 Chocolatey 社区自行打包。每个包的下载 URL、校验和由社区维护的包描述文件记录，`choco install` 会自动验证数字签名（如果上游提供）。

### 配置编译环境

安装 West 工具：

```bash
pip install west
```

> **这条命令做了什么：** 从 Python 官方包仓库 [PyPI](https://pypi.org/project/west/) 下载并安装 `west` 命令行工具。West 是 Zephyr RTOS 项目的官方元工具（meta-tool），底层调用 Git 和 CMake 来管理多仓库工程。安装位置在 Python 的 `Scripts\` 目录下（如 `C:\Python311\Scripts\west.exe`），不会修改系统服务或注册表。简单理解：它就是 Zephyr 世界的 `repo` + `cmake` 统一入口。

获取 Chipsea Zephyr 工程包（无需从 GitHub 克隆，直接使用提供的 SDK）。目录结构：

```
EC-ZEPHYR
┣ .vscode
┗ .west
  ┗ ecfw-zephyr
  ┗ ecfwwork
    ┣ modules
    ┗ zephyr_fork
    ┗ zephyr_modules_csce250x
```

> **这笔操作做了什么：** 直接解压芯海提供的 SDK 压缩包到你的工作目录。与在线 `west init && west update` 不同，这里使用的是**本地离线 SDK**，不会连接任何远程仓库，不依赖网络。目录关系：
> - `ecfw-zephyr/` — EC 固件主工程（应用代码 + 板级配置），是你日常开发的工作区
> - `ecfwwork/zephyr_fork/` — Zephyr RTOS 内核代码（上游 fork，包含内核、子系统、驱动框架）
> - `ecfwwork/zephyr_modules_csce250x/` — 芯海芯片 HAL 层（片上外设驱动、SoC 配置文件）
> - `ecfwwork/modules/` — Zephyr 外部模块（MCUboot、TinyCBOR 等）
> - `.west/config` — west 配置文件，记录各个仓库的本地路径映射关系

导出 Zephyr CMake 包：

```bash
west zephyr-export
```

> **这条命令做了什么：** 在 Windows 注册表中向 CMake 注册 Zephyr 包的位置。具体写入路径为 `HKEY_CURRENT_USER\Software\Kitware\CMake\Packages\Zephyr`，值是 Zephyr CMake 配置文件的路径。这样后续 `cmake` 命令可以通过 `find_package(Zephyr)` 自动定位 Zephyr SDK 的位置，无需手动设置环境变量。这条命令**只写入当前用户的注册表**（HKCU），不影响系统全局（HKLM），也不会修改任何文件。

成功日志示例：

```
Zephyr (E:/common/ec/EC-Zephyr/ecfwwork/zephyr_fork/share/zephyr-package/cmake)
has been added to the user package registry in:
HKEY_CURRENT_USER\Software\Kitware\CMake\Packages\Zephyr
```

### 安装 Python 依赖

```bash
pip install -r ecfwwork\zephyr_fork\scripts\requirements.txt
```

> **这条命令做了什么：** 根据 Zephyr 官方维护的 `requirements.txt` 清单，批量安装编译 Zephyr 固件所需的 Python 依赖包。主要包括：
> - `pyelftools` — 解析 ELF 格式固件文件，用于生成内存使用报告
> - `pykwalify` / `PyYAML` — 解析 YAML 设备树绑定文件和 Kconfig 配置
> - `west` 自身的依赖（如果已安装则跳过）
> - `canopen` — CAN 总线协议栈 Python 工具
> - 其他辅助工具（构建脚本、代码检查、测试框架等）
>
> 这些包全部来自 PyPI 官方仓库，安装到 Python 的 `Lib\site-packages\` 目录，不会修改系统环境。**安全提示：** 可以先查看 `requirements.txt` 内容确认依赖列表（每个包的版本都已固定），再执行安装。

### 配置交叉编译链（Zephyr SDK）

使用提供的 `zephyr-sdk-0.16.8` 包：

1. 解压到 C 盘（或其他盘）
2. 进入 `zephyr-sdk-0.16.8` 目录，双击执行 `setup.cmd`
3. 仅安装 Host tools + 注册 Zephyr SDK CMake 包，其余选项选 N
4. 显示 "All done" 即配置成功

> **`setup.cmd` 做了什么：**
> | 选项 | 默认 | 作用 | 建议 |
> |------|------|------|------|
> | Host tools | 安装 | 安装 OpenOCD 调试桥接、QEMU 模拟器等主机端工具 | ✅ 必选 |
> | Register CMake package | 安装 | 向 CMake 注册 Zephyr SDK 的编译工具链路径（写入注册表 `HKCU\Software\Kitware\CMake\Packages\Zephyr-SDK`） | ✅ 必选 |
> | ARM toolchain | N | 安装 `arm-zephyr-eabi` 交叉编译器（~200 MB），用于编译 ARM 架构固件 | ❌ 芯海 SDK 已自带交叉编译器，无需安装 |
> | RISC-V / x86 / xtensa / ... | N | 其他架构的交叉编译器 | ❌ 本工程只用到 ARM 交叉编译器 |
>
> **为什么大部分选项选 N：** 芯海 SDK 包里已包含编译所需的 `arm-zephyr-eabi-gcc` 交叉编译器，与 Zephyr SDK 中的版本完全一致。重复安装只是浪费 ~2 GB 磁盘空间，还可能造成版本冲突。`setup.cmd` 的所有写入操作都限制在 `zephyr-sdk-0.16.8\` 目录和当前用户注册表内，不会修改系统文件。

![](/media/image2.png)

> 确认未设置 `ZEPHYR_TOOLCHAIN_VARIANT` 环境变量。Chipsea EC Zephyr 项目默认使用 Zephyr SDK 编译，编译脚本已自动忽略该变量。

### 首次编译

在 `ecfw-zephyr` 路径下执行：

```batch
build.bat
```

> **这条命令做了什么：** 执行芯海 EC 工程自带的编译脚本。`build.bat` 内部实际调用的是 `west build -b <board_name> . --build-dir build/ec_fw`，等价于以下自动化步骤：
> 1. **Kconfig 解析** — 根据 `app/csce250x/Kconfig` 和板级 `prj.conf` 生成 `.config`，确定哪些 EC 功能模块参与编译
> 2. **设备树编译** — 将板级 `.dts` 文件编译为二进制 `.dtb`，生成 `devicetree_generated.h` 头文件供 C 代码引用
> 3. **CMake 构建系统生成** — 生成 Ninja 构建文件
> 4. **源码编译** — `arm-zephyr-eabi-gcc` 交叉编译所有 C/ASM 源文件 → 目标文件（`.o`）
> 5. **链接** — 将所有 `.o` 按 Flash/RAM 布局链接为 ELF 文件
> 6. **转换** — ELF → HEX（用于烧录）→ BIN（用于 OTA）
> 7. **报告** — 输出 Flash/RAM 使用量统计
>
> 编译产物全部输出到 `build/ec_fw/` 目录：
> - `APROM.hex` — Flash 烧录镜像
> - `zephyr.elf` — 带调试符号的 ELF 文件
> - `zephyr.bin` — 原始二进制镜像（用于 OTA 升级）
>
> **安全说明：** `build.bat` 是本地脚本，只读取工程源码、调用本地安装的交叉编译器。整个过程**不会连接网络**，不会上传任何代码或产物。

### 烧录测试

1. 安装 J-Link 开发包
2. 导入 Chipsea 芯片开发包
3. 打开 J-Flash，选择对应芯片设备（如 CSCE250X）
4. 导入 `build\ec_fw\APROM.hex`
5. 烧录成功后，打开 J-Link RTT Viewer 即可看到 Zephyr 运行日志

> **这些步骤做了什么：**
> - **J-Link 开发包** — SEGGER 公司提供的 SWD 调试器驱动和上位机软件（J-Flash、J-Link RTT Viewer），来自 [segger.com](https://www.segger.com/downloads/jlink/)。安装时会添加 USB 驱动，使 J-Link 调试器能被系统识别。写入位置在 `C:\Program Files\SEGGER\`。
> - **Chipsea 芯片开发包** — 芯海提供的 J-Flash 芯片支持文件，包含 CSCE2010/250x/2520 的 Flash 算法和芯片描述，使 J-Flash 能识别芯海 MCU。这部分由芯海技术支持提供，是离线文件。
> - **J-Flash 烧录** — 通过 SWD 接口将 `APROM.hex` 写入 MCU 的 Flash 存储。烧录过程完全由 J-Flash 控制，只访问你的本地调试器 USB 设备和目标 MCU。
> - **J-Link RTT Viewer** — 通过 SWD 接口读取 MCU 运行时输出的文本日志（RTT = Real-Time Transfer），不占用 UART 串口，即使 MCU 不引 UART 引脚也能看到日志。

## Linux 平台环境搭建

基于 Zephyr 3.7.0 LTS，支持 Ubuntu（20.04+）、Fedora、Clear Linux 及 Arch Linux。

### 系统更新与基础依赖

```bash
# Ubuntu / Debian
sudo apt update && sudo apt upgrade

# Fedora
sudo dnf upgrade

# Arch Linux
sudo pacman -Syu
```

> **这些命令做了什么：**
> - `sudo apt update` — 从 `/etc/apt/sources.list` 中配置的软件源服务器（通常是各 Linux 发行版的官方镜像）下载最新的软件包索引文件到本地缓存（`/var/lib/apt/lists/`）。只更新「目录」，不安装或升级任何软件。
> - `sudo apt upgrade` — 根据最新索引，将已安装的软件包升级到最新版本。会修改系统文件，但只替换有更新的包，不会删除或新增软件。
> - `sudo dnf upgrade` (Fedora) / `sudo pacman -Syu` (Arch) — 等价操作，先刷新源索引再升级系统包。
>
> **安全说明：** 这些都是 Linux 发行版的标准系统维护命令，不涉及第三方源。如果你的组织使用内部镜像源，`apt update` 会从内部源获取索引。

**Ubuntu 22.04+：**

```bash
sudo apt install --no-install-recommends git cmake ninja-build gperf \
  ccache dfu-util device-tree-compiler wget \
  python3-dev python3-pip python3-setuptools python3-tk python3-wheel xz-utils file \
  make gcc gcc-multilib g++-multilib libsdl2-dev libmagic1
```

> **这条命令做了什么 —— 每个包的作用：**
>
> | 包名 | 用途 | 类别 |
> |------|------|------|
> | `git` | 版本控制工具，west 底层依赖它管理多仓库 | 构建工具链 |
> | `cmake` | C/C++ 构建系统，Zephyr 的编译入口 | 构建工具链 |
> | `ninja-build` | 快速构建执行器，CMake 调用它执行实际编译 | 构建工具链 |
> | `gperf` | 完美哈希函数生成器，Zephyr 构建过程用来生成设备树查找表 | 构建工具链 |
> | `ccache` | 编译缓存工具，缓存 `.o` 文件避免重复编译（后续增量编译可提速 5-10 倍） | 效率工具 |
> | `dfu-util` | USB DFU（Device Firmware Upgrade）烧录工具 | 烧录工具 |
> | `device-tree-compiler` | 设备树编译器（`dtc`），将 `.dts` 源文件编译为 `.dtb` 二进制 | 构建工具链 |
> | `wget` | 命令行文件下载工具 | 辅助工具 |
> | `python3-dev` | Python 开发头文件（`Python.h`），部分 Python 包（pyelftools）编译时需要 | Python 工具链 |
> | `python3-pip` | Python 包管理器，安装 `west` 和其他依赖 | Python 工具链 |
> | `python3-setuptools` | Python 包构建工具 | Python 工具链 |
> | `python3-tk` | Python Tkinter GUI 库，部分 Zephyr 工具（menuconfig）需要 | Python 工具链 |
> | `python3-wheel` | Python wheel 打包格式支持 | Python 工具链 |
> | `xz-utils` | XZ 压缩格式解压工具，SDK 包使用 `.tar.xz` 格式 | 辅助工具 |
> | `file` | 文件类型识别工具 | 辅助工具 |
> | `make` | GNU Make，部分子模块可能用到 | 构建工具链 |
> | `gcc` / `gcc-multilib` | GNU C 编译器 + 32 位库支持（Zephyr SDK 的部分 host 工具需要 32 位运行时） | 编译环境 |
> | `g++-multilib` | GNU C++ 编译器 32 位库支持 | 编译环境 |
> | `libsdl2-dev` | SDL2 图形库开发包，Zephyr 模拟器（QEMU）和 GUI 工具需要 | 模拟器 |
> | `libmagic1` | 文件类型魔法数字检测库，构建脚本用于识别文件格式 | 辅助工具 |
>
> **安全说明：** `--no-install-recommends` 表示只安装**必须**的依赖，跳过"建议安装"的附加包，减少系统修改面。所有包从 Ubuntu 官方 APT 仓库安装，数字签名由 GPG 密钥链自动验证。

> Ubuntu < 22.04 需先添加 Kitware APT 仓库：`wget https://apt.kitware.com/kitware-archive.sh && sudo bash kitware-archive.sh`

**Fedora：**

```bash
sudo dnf group install "Development Tools" "C Development Tools and Libraries"
sudo dnf install cmake ninja-build gperf dfu-util dtc wget which \
  python3-pip python3-tkinter xz file python3-devel SDL2-devel
```

> 与 Ubuntu 端安装的包功能相同，只是包名在 Fedora 下略有不同（如 `python3-devel` 替代 `python3-dev`、`dtc` 替代 `device-tree-compiler`）。所有包从 Fedora 官方 `dnf` 仓库安装，由 RPM 签名验证。

**Arch Linux：**

```bash
sudo pacman -S git cmake ninja gperf ccache dfu-util dtc wget \
  python-pip python-setuptools python-wheel tk xz file make
```

> 与 Ubuntu 端功能相同。Arch 下 `python-pip` 替代 `python3-pip`，`dtc` 替代 `device-tree-compiler`。所有包从 Arch 官方仓库安装，由 pacman 密钥环验证。

验证版本：`cmake --version`、`python3 --version`、`dtc --version`（要求 CMake ≥ 3.20.5、Python ≥ 3.10、DTC ≥ 1.4.6）。

> **验证命令做了什么：** 每条都是纯查询——只读取已安装工具的版本号并打印到终端，不修改任何文件或配置。如果版本不满足要求，说明系统包版本过旧，需要按发行版提示添加新版源或手动升级。

### Python 虚拟环境

```bash
sudo apt install python3-venv
python3 -m venv ~/ec-zephyr-env
source ~/ec-zephyr-env/bin/activate
```

> **这三条命令做了什么：**
> 1. `sudo apt install python3-venv` — 安装 Python 标准库中的 `venv` 模块（Ubuntu 将它与 Python 本体分开打包）。只安装这一个包。
> 2. `python3 -m venv ~/ec-zephyr-env` — 在你的 Home 目录下创建一个名为 `ec-zephyr-env` 的虚拟环境目录。这个目录包含一个独立的 Python 解释器副本和 `lib/python3.x/site-packages/` 目录，大约 20 MB。**关键好处**：后续所有 `pip install` 都安装到这个独立目录内，不会污染系统 Python 环境，也不会与系统已安装的 Python 包冲突。
> 3. `source ~/ec-zephyr-env/bin/activate` — 修改当前 Shell 会话的 `$PATH` 和 `$VIRTUAL_ENV` 环境变量，使 `python` 和 `pip` 命令指向虚拟环境中的版本。
>
> **安全说明：** 虚拟环境完全是你用户空间下的一个目录，不涉及系统文件或注册表。如果不需要了，直接 `rm -rf ~/ec-zephyr-env` 即可彻底清除，不回留任何痕迹。

> 每次新开终端需重新激活：`source ~/ec-zephyr-env/bin/activate`。退出使用 `deactivate`。

### 配置交叉编译链（Zephyr SDK 0.16.8）

```bash
# 下载（x86_64）
cd ~
wget https://github.com/zephyrproject-rtos/sdk-ng/releases/download/v0.16.8/zephyr-sdk-0.16.8_linux-x86_64.tar.xz
```

> **这条命令做了什么：** 从 GitHub 上的 Zephyr 官方仓库下载 Zephyr SDK 压缩包。下载源为 `github.com/zephyrproject-rtos/sdk-ng`，由 Linux 基金会 Zephyr 项目维护，不是第三方源。压缩包约 1.5 GB，包含 ARM、RISC-V 等多架构交叉编译器和你本机 Linux 的编译工具链。保存位置是你当前目录（`~/`）。

```bash
# 验证完整性
wget -O - https://github.com/zephyrproject-rtos/sdk-ng/releases/download/v0.16.8/sha256.sum | shasum --check --ignore-missing
```

> **这条命令做了什么：** 这是**安全验证**环节，两步合一：
> 1. `wget -O - <url>` — 下载官方发布的 SHA256 校验和文件到内存（不写磁盘），通过管道输出
> 2. `shasum --check --ignore-missing` — 对当前目录下的 `zephyr-sdk-0.16.8_linux-x86_64.tar.xz` 计算 SHA256 哈希值，与官方校验文件比对。`--ignore-missing` 表示只检查已下载的文件，忽略校验文件中列出的其他架构文件。
>
> 如果输出 `zephyr-sdk-0.16.8_linux-x86_64.tar.xz: OK` 表示文件完整且未被篡改。如果输出 `FAILED`，说明下载不完整或文件已被改动，**应立即删除重新下载**，不要继续后续步骤。

```bash
# 解压
tar xvf zephyr-sdk-0.16.8_linux-x86_64.tar.xz
```

> **这条命令做了什么：** 将 SDK 压缩包解压到 `~/zephyr-sdk-0.16.8/` 目录。这是本地文件操作，不依赖网络。解压后约占用 5 GB 磁盘空间。

```bash
# 安装
cd zephyr-sdk-0.16.8
./setup.sh
```

> **这条命令做了什么：** 执行 Zephyr SDK 的安装脚本。它在你的 `~/.zephyrrc` 文件中写入环境变量，并向 CMake 的用户包注册表中注册 Zephyr SDK 路径（`~/.cmake/packages/Zephyr-SDK/`）。**不会**执行 `sudo`，所有修改都限于你的用户目录，不影响系统文件。与 Windows 的 `setup.cmd` 同理，ARM 交叉编译器选项同样无需安装（芯海 SDK 已自带）。

```bash
# 安装 udev 规则（允许普通用户烧录）
sudo cp ~/zephyr-sdk-0.16.8/sysroots/x86_64-pokysdk-linux/usr/share/openocd/contrib/60-openocd.rules /etc/udev/rules.d
sudo udevadm control --reload
```

> **这两条命令做了什么：**
> 1. `sudo cp ... /etc/udev/rules.d` — 将 OpenOCD（调试桥接软件）的 udev 规则文件复制到系统 udev 规则目录。这条规则告诉 Linux：当 USB 插入 J-Link 或其他调试器时，自动将设备文件的读写权限开放给 `plugdev` 用户组。**这是本 Linux 安装指南中唯一需要 `sudo` 写入系统文件的操作。** 源文件来自 Zephyr SDK 包内，你可以先查看内容再执行：`cat ~/zephyr-sdk-0.16.8/sysroots/*/usr/share/openocd/contrib/60-openocd.rules`
> 2. `sudo udevadm control --reload` — 通知 udev 守护进程重新加载规则文件，使新规则立即生效，无需重启。

### 环境变量

```bash
unset ZEPHYR_TOOLCHAIN_VARIANT                                    # 清除冲突变量
export ZEPHYR_SDK_INSTALL_DIR=/path/to/zephyr-sdk-0.16.8         # 非默认路径时设置
```

> **这两条命令做了什么：**
> 1. `unset ZEPHYR_TOOLCHAIN_VARIANT` — 清除 Shell 中可能已存在的 `ZEPHYR_TOOLCHAIN_VARIANT` 环境变量（有些用户之前用过其他 Zephyr 项目设置了这个变量）。芯海 EC 工程的 CMake 脚本会自动选择交叉编译器，如果这个变量被设置会干扰自动选择逻辑。`unset` 只影响当前 Shell 会话的内存环境，不写任何文件。
> 2. `export ZEPHYR_SDK_INSTALL_DIR` — 如果你的 Zephyr SDK 安装路径不是默认的 `~/zephyr-sdk-0.16.8`（比如解压到了 `/opt/`），用这个变量告诉 CMake 去哪里找 SDK。如果你按本指南操作（解压到 `~/`），无需设置这一行。`export` 也只影响当前 Shell 会话，不写文件。

可追加到 `~/.bashrc` 或 `~/.zshrc` 持久化。

### 烧录测试

1. 安装 J-Link 软件包（[SEGGER 官网](https://www.segger.com/downloads/jlink/)）
2. 安装 Chipsea 芯片开发包
3. 启动 J-Flash 或 JLinkExe，选择对应芯片设备（如 CSCE250X）
4. 加载 `build/ec_fw/APROM.hex`
5. 烧录后通过 J-Link RTT Viewer 查看运行日志

> **这些步骤做了什么（Linux 特化说明）：**
> - **J-Link 软件包** — SEGGER 提供 `.deb`/`.rpm` 安装包，安装到 `/opt/SEGGER/` 目录，通过 `dpkg` 或 `rpm` 管理，可以随时卸载。安装时会添加 udev 规则使调试器 USB 设备可访问。
> - **Chipsea 芯片开发包** — 与 Windows 端相同，是芯海提供的本地文件，离线安装到 J-Link 的设备目录中（通常为 `/opt/SEGGER/JLink/Devices/Chipsea/`）。
> - **JLinkExe / J-Flash** — 通过 USB 连接的 SWD 调试器与目标 MCU 通信，将 HEX 固件写入 Flash。烧录过程中调试器本身通过 USB 供电和通信，不涉及网络。
> - **J-Link RTT Viewer** — 通过 SWD 接口实时读取 MCU 运行日志（与 UART 串口无关），验证固件成功启动。

### 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| `cmake --version` 版本过低 | 系统 CMake < 3.20.5 | 添加 Kitware APT 仓库或从官网安装新版 |
| `west: command not found` | pip 路径不在 `$PATH` | `export PATH=$HOME/.local/bin:$PATH` |
| SDK `./setup.sh` 失败 | 缺少 32 位库 | 安装 `gcc-multilib` 和 `g++-multilib` |
| 烧录权限不足 | 未安装 udev 规则 | 安装 `60-openocd.rules` |
| 虚拟环境未激活 | 新终端未 source | 执行 `source ~/ec-zephyr-env/bin/activate` |
