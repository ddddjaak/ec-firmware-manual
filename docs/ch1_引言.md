# 引言


## 文档目标与适用对象

本开发指南旨在帮助 现场应用工程师 (FAE) 和 客户应用开发者
快速理解并掌握基于 Zephyr RTOS 的嵌入式控制器 (EC) 固件开发方法。

通过阅读本文档，读者可以：

- 了解 EC 在系统中的角色和典型应用场景；

- 熟悉基于 OpenEC 框架 与 Zephyr RTOS 的固件开发流程；

- 掌握固件的构建、裁剪、移植、调试与问题定位方法；

- 能够基于现有框架实现 自定义功能扩展，满足产品差异化需求。

本指南主要适用于以下读者：

- FAE（现场应用工程师）：需要快速搭建演示环境，验证功能，并为客户提供技术支持。
  FAE 通常需要在客户现场完成 `hello_world` 级别的最小系统验证，并演示键盘响应、
  LED 控制等基础外设功能。建议优先阅读 第 2 章（环境搭建）和第 4 章（核心功能模块），
  快速跑通构建流程后再按需深入。

- 客户固件开发者：希望在已有 OpenEC 基础上，进行二次开发或扩展功能。
  开发者需要重点理解 EC 各子系统的 API 调用方式、驱动模型与设备树绑定规则，
  以及如何编写自定义功能模块并集成到现有框架中。建议按顺序通读
  第 2 章 → 第 3 章 → 第 4 章 → 第 5 章，建立起完整的开发闭环。

- QA / 测试工程师：需要根据 EC 功能规范编写自动化测试用例，进行回归验证和压力测试。
  Zephyr 内置的 Twister 测试框架支持在 `native_sim` 平台和真实硬件上运行测试集，
  测试工程师应重点参考 第 6 章（调试验证）以及各功能模块章节中的测试说明部分。

- 硬件 / 基板设计工程师：需要根据 EC 固件需求确认 Pinmux 配置、I²C/SPI 总线的设备树
  描述，以及 GPIO、ADC 通道的分配方案。Zephyr 使用 Devicetree 描述硬件拓扑，
  硬件工程师应重点阅读 第 7 章（移植与定制）中的设备树定制章节，确保硬件设计与
  固件配置对齐。

- 研发团队成员：对 EC 固件架构与调试手段有学习需求，可作为团队内部知识库参考。

## 什么是嵌入式控制器

嵌入式控制器（Embedded Controller,
EC）通常指存在于笔记本电脑、服务器、工业计算机或嵌入式系统中的一颗低功耗微控制器，其主要任务是：

- 管理 电源序列与状态切换（如 S0/S3/S5 电源状态控制）；

- 处理 键盘、触控板等输入外设；

- 实现 电池与充电控制，支持智能电源管理；

- 控制 风扇、热传感器 等，实现热管理；

- 提供与 主机 CPU/SoC 的通信通道（如 LPC/eSPI/I²C 等）。

在现代计算设备中，EC 已成为系统稳定运行和用户体验优化的重要组成部分。

### EC 与其他处理器的对比

EC 的定位介于通用 MCU 与桌面级 CPU 之间，在设计目标上存在显著差异：

| 维度 | 嵌入式控制器 (EC) | 通用 MCU | 桌面级 CPU |
|---|---|---|---|
| 功耗 | 超低功耗 (mW 级) | 低功耗 | 高功耗 (数十 W) |
| 实时性 | 强实时 (sub-ms 中断响应) | 实时/非实时混合 | 非实时 |
| 外设集成 | 高度专用 (键盘扫描、电池管理、风扇 PWM) | 通用外设 | 外部芯片组配合 |
| 可靠性要求 | 极高 (always-on, 内置看门狗与双 Bank 固件) | 中高 | 中 |
| 供电域 | 5V/3.3V 双域支持，需在 S5/G3 状态下保持部分电路存活 | 单域 | 复杂多域电源 |
| 典型核心 | Cortex-M33 (如 CSCE250X) | Cortex-M0/M3/M4/M33 | x86-64 / ARM A-series |

### EC 的特殊设计约束

与通用嵌入式系统不同，EC 面临一系列特有的设计约束：

- Always-On 特性：EC 必须在系统 S5（软关机）甚至 G3（机械断电）后仍保持部分功能运行，
  例如 RTC 计时、电池充电管理、电源按钮监听等。这要求 EC 固件具备极低功耗的待机路径。

- 双域供电 (5V/3.3V)：主板在运行时提供 3.3V 主供电，但在深度睡眠或关机状态下
  仅保留 5V 待机电源。EC 芯片需同时支持两个供电域，固件也需要在供电域切换时正确
  保存和恢复状态。

- 亚毫秒级中断响应：键盘扫描、eSPI 主机命令、电源序列控制等场景要求 EC 在
  sub-ms 级别完成中断响应和处理，否则会出现按键丢失或电源时序异常等严重问题。

- 防砖机制（双 Bank Flash）：EC 固件升级过程中若发生掉电，可能导致设备变砖。
  OpenEC 采用的 CSCE250X（Cortex-M33 内核）支持双 Bank Flash 架构，可通过 A/B 分区
  实现安全 OTA，升级失败时自动回滚到上一版本，确保设备始终可启动。

### CSCE250X 简介

本开发指南以 CSCE250X（Cortex-M33 内核，Armv8-M 架构）为目标平台。该芯片专为 EC
场景设计，集成了键盘扫描矩阵、多通道 PWM、高精度 ADC、eSPI 从控制器等 EC 专属外设，
并支持 TrustZone 安全扩展与双 Bank Flash 安全升级。OpenEC 框架已为其提供了完整的
Zephyr 板级支持包 (BSP)。

## 基于 OpenEC + Zephyr 的开发优势

传统 EC 固件多采用 闭源 RTOS 或厂商自研框架，存在
学习曲线陡峭、生态支持不足、扩展性差 等问题。基于 OpenEC 框架 + Zephyr
RTOS 的开发方案，则具备以下优势：

- 开源与社区支持

<!-- -->

- Zephyr 由 Linux Foundation 主导，拥有活跃的全球开发者社区；

- 开源代码透明，便于客户定制与快速迭代；

- OpenEC 复用了 Zephyr 上游的大量成熟模块，包括：
  - `logging`：结构化日志子系统，支持多后端输出（UART、RTT、RAM 缓冲区）；
  - `shell`：交互式命令行，可在运行时查询寄存器、修改配置、触发测试；
  - `zbus`：轻量级消息总线，用于 EC 各子系统间的事件驱动通信；
  - `settings`：基于 Flash 的持久化存储，支持系统配置的读写与恢复出厂设置；
  - `SMP (Simple Management Protocol)`：支持通过 UART/USB 进行远程固件管理
    和 OTA 升级；
  - 这些模块均经过社区长期维护和测试，降低了 OpenEC 的自研成本和维护负担。

<!-- -->

- 模块化与可裁剪性

<!-- -->

- Zephyr 提供丰富的设备驱动与中间件，可根据硬件平台裁剪；

- OpenEC 封装了典型的 EC 功能模块（电源、键盘、风扇等），降低开发难度；

- Kconfig 配置系统提供超过 1000 个配置选项，以树状依赖关系组织：
  - 启用高层功能（如 `CONFIG_EC_POWER_MGMT`）会自动选中所需的下层依赖（驱动、
    子系统、库），无需开发者手动追踪依赖链；
  - 从 KB 级裸机应用到 MB 级全功能 EC 固件，均可通过 Kconfig 精确控制二进制
    体积，实现按需裁剪；
  - 每个功能模块提供独立的 Kconfig 开关和帮助文档，开发者可通过 `menuconfig`
    或 `guiconfig` 图形化工具直观地进行配置。

<!-- -->

- 跨平台与标准化

<!-- -->

- 支持多种 MCU 架构（ARM Cortex-M, RISC-V 等）；

<!-- -->

- 统一的设备树和 Kconfig 配置机制，降低移植成本；

- 同一份 EC 固件代码可编译运行于三种目标：
  - `native_sim`：在 Linux 主机上原生执行，支持 GDB 断点调试和 AddressSanitizer /
    UndefinedBehaviorSanitizer 运行时检测，是 CI 自动化测试的首选平台；
  - `QEMU`：ARM 指令级仿真，无需真实硬件即可验证外设驱动的逻辑正确性；
  - 真实 ARM 硬件（CSCE250X 开发板）：完整的功能验证与性能测试；

- Zephyr 的设备驱动 API 在不同芯片厂商之间保持统一，例如 `i2c_transfer()`、
  `gpio_pin_set()` 等接口与底层硬件无关，切换芯片平台时上层业务代码无需修改，
  仅需调整设备树配置。

<!-- -->

- 工程效率与可维护性

<!-- -->

- 内置日志、Shell、调试接口，方便快速定位问题；

- 标准化的 API 接口与任务模型，便于团队协作与后续维护；

- 完善的自动化测试与质量保障工具链：
  - Twister：Zephyr 官方测试框架，支持在 `native_sim`、QEMU 和真实硬件上
    批量运行数千个测试用例，自动生成测试报告，可集成到 CI/CD 流水线；
  - Sanitizer 支持：在 `native_sim` 平台上可启用 `AddressSanitizer (ASan)`
    和 `UndefinedBehaviorSanitizer (UBSan)`，在开发阶段即捕获内存越界、
    使用后释放、整数溢出等问题；
  - 静态分析：支持 Coverity Scan 和 Clang Static Analyzer 集成，在代码
    合入前自动识别潜在缺陷；
  - 代码体积跟踪：`west build -t ram_report` 和 `west build -t rom_report`
    命令可生成详细的 RAM/ROM 占用报告，精确到每个符号和源文件，方便进行
    体积优化。

## EC 固件开发流程概览

EC 固件的开发流程可概括为以下四个阶段：

1.  **环境搭建** — 安装 Zephyr SDK、west 工具链、Python 依赖等，获取 Chipsea Zephyr 工程包。详见第 2 章「开发环境准备」。

2.  **代码开发** — 基于 `ecfw-zephyr/app/` 中的功能模块和 `drivers/` 中的驱动进行开发。通过 Kconfig 配置功能开关，通过 Devicetree 描述硬件拓扑。详见第 3 章「架构概述」和第 4 章「固件功能模块说明」。

3.  **构建与烧录** — 使用 `build.bat`（Windows）或 `west build` 命令编译固件，生成 `APROM.hex`。通过 J-Link 或 DediProg 烧录到目标芯片。详见第 6 章「调试与验证」。

4.  **调试与测试** — 通过 J-Link RTT Viewer 查看运行日志，使用 Shell 命令交互调试，运行 twister 单元测试验证功能。详见第 6 章「调试与验证」。

> **参考**：Chromium OS 的 EC 构建流程（使用 chroot + `make BOARD=<name>`）可参考 [Chromium OS Developer Guide](https://chromium.googlesource.com/chromiumos/docs/+/HEAD/developer_guide.md)，但 Chipsea Zephyr EC 项目使用独立的工程包和构建脚本，不依赖 Chromium OS 环境。
